# 業務要件一覧

| id | category | requirement | priority | description |
| --- | --- | --- | --- | --- |
| BR-001 | 運用管理 | 常駐処理と排他制御の実施 | 高 | システム運用上、単一のプロセスでのみ継続的に業務処理を行う必要があるため、アプリケーションは常駐処理として稼働し、同一環境内での重複起動を防ぐための排他制御を行うこと。 |
| BR-002 | ホスト管理 | 静的ホストの管理と維持 | 高 | 運用担当者が監視対象のネットワーク機器を管理できるよう、管理画面（CLI）から「静的ホストの追加」「静的ホストの削除」「静的ホストの表示」を行うことができ、設定された情報をstatic_hostsとして保持すること。 |
| BR-003 | 名前解決 | 名前解決の実行と自己レコードの生成 | 高 | 管理対象として登録されたstatic_hostsに対して定期的に名前解決業務を実施し、取得したIPアドレス等の結果を自拠点の情報（self_records）として登録・更新すること。 |
| BR-004 | 名前解決 | 自己参照ループの回避 | 中 | 誤ったキャッシュ情報や自身の過去の応答情報を再登録してしまう自己参照ループによる業務影響を防ぐため、名前解決時には実在確認を伴う特定の手法を用いること。また、ループバックアドレスは登録対象外とすること。 |
| BR-005 | レコード連携 | プロキシ発見とレコード同期 | 高 | 複数環境間での情報を統合するため、ネットワーク内の他のノードを探索（プロキシ発見）し、自身が持つself_recordsと有効な外部情報を他ノードへ送信（レコード同期）すること。外部からの情報は「/api/other-records」経由で受信し、other_recordsとして管理すること。 |
| BR-006 | レコード管理 | マージ処理の実行 | 高 | 自拠点で解決したself_recordsと他拠点から受信したother_recordsの情報を統合し、ネットワーク全体における最新の名前解決結果としてmerged_recordsを生成・更新するマージ処理を定期的に行うこと。 |
| BR-007 | レコード管理 | レコードの有効期限管理 | 高 | 保持しているself_recordsおよびother_recordsの鮮度を保つため、各情報に設定された有効期限（TTL）を定期的に減算し、期限切れとなった古い情報はシステムから除外すること。 |
| BR-008 | 外部提供機能 | マルチキャストグループでの情報応答 | 高 | ネットワーク内のクライアント機器からの問い合わせに対応するため、マルチキャストグループに参加して要求を受信し、統合されたmerged_recordsの情報を基に適切なアドレス情報を応答すること。 |
| BR-009 | 外部連携 | マージ済みレコードのAPI提供 | 中 | 外部の連携システムや運用監視ツールが現在のネットワーク解決状況を参照できるよう、「/api/merged-records」を通じて最新のmerged_recordsの一覧データを提供すること。 |
| BR-010 | 運用保守 | 状況確認とインスタント実行モードの提供 | 低 | 運用担当者がトラブルシューティングや即時反映を行うための業務機能として、「マージ済みレコードの表示」機能を提供し、さらに常駐処理を伴わずにプロキシ発見やマージ処理等を1回限り実行する「インスタント実行モード」を提供すること。 |
| BR-011 | データ管理 | マイグレーションによる設定の引継ぎ | 中 | 業務に影響を与えずにシステムの仕様変更へ対応するため、起動時にマイグレーション処理を実施し、既存のstatic_hosts等の管理データを安全に新しいデータ構造へ移行すること。 |
